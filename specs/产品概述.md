# 产品概述

## 1. 项目愿景 (Vision)
*用一句话概括：我们要为谁，提供什么核心价值？*
> 为个人开发者提供一套功能统一、使用习惯一致的跨平台开发常用工具集合，解决不同平台开发工具功能不一致的痛点，提高开发效率。

## 2. 核心价值 (Value Proposition)
*   **用户价值**：解决跨平台开发工具功能不一致的问题，提供统一界面和操作习惯，减少学习成本，提升开发效率
*   **商业/技术价值**：基于 Electron 构建，一次开发双平台运行，架构设计具备可扩展性，便于后续添加新工具

## 3. 核心板块 (Core Modules)
*将系统拆解为关键的顶层模块（建议 2-4 个核心板块）。*

### 3.1 编码解码工具包
*   **描述**：提供常用编码解码功能的工具集合
*   **关键特性**：
    *   Base64 编码/解码
    *   URL 编码/解码
    *   JSON 格式化/压缩

### 3.2 网络工具包
*   **描述**：提供网络开发调试相关的工具
*   **关键特性**：
    *   HTTP 请求测试（支持 GET/POST/PUT/DELETE 等方法）
    *   WebSocket 调试（连接、发送消息、接收消息）

### 3.3 数据处理工具包
*   **描述**：提供常用数据转换和处理功能
*   **关键特性**：
    *   时间戳转换（时间戳↔日期时间）
    *   正则表达式测试（匹配、替换、分组提取）

### 3.4 历史记录管理
*   **描述**：统一管理各类工具的使用历史
*   **关键特性**：
    *   HTTP 请求历史保存与重用
    *   常用正则表达式收藏与管理
    *   各类转换工具的历史记录

## 4. 目标用户 & 核心场景 (User & Scenes)
### 4.1 目标用户
*   **个人开发者**：在 Windows 和 macOS 之间切换工作的开发者，需要一套功能一致的开发辅助工具

### 4.2 核心场景 (Key Use Cases)
*   **场景 1**：开发者在调试 API 时，通过 HTTP 请求测试工具发送请求，保存常用请求到历史记录，下次直接重用。
*   **场景 2**：开发者在处理前端数据时，需要快速格式化 JSON 数据，使用编码解码工具包的 JSON 格式化功能，并查看历史记录中的格式化结果。
*   **场景 3**：开发者在开发 WebSocket 应用时，使用 WebSocket 调试工具连接服务器，发送测试消息，查看返回结果。
*   **场景 4**：开发者在处理日志数据时，使用正则表达式测试工具调试匹配规则，保存常用正则表达式以便后续使用。

## 5. 关键业务流程 (Core Workflows)
*描述系统最核心的 1-2 条业务链路，这决定了技术架构的复杂度。*
> 工具调用流程：用户选择工具 → 输入数据 → 执行操作 → 显示结果 → 可选保存到历史记录

## 6. 验收标准 (Acceptance Criteria)
*项目第一阶段结束的具体标志是什么？（用于防止需求蔓延）*
*   **功能完备性**：所有计划的工具（编码解码、网络工具、数据处理）均正常工作，历史记录功能可用
*   **技术指标**：应用启动时间 < 3s，工具操作响应 < 500ms，Windows 和 macOS 平台均能正常运行

## 7. 约束条件 (Constraints)
*非功能性的硬性限制。*
*   使用 Electron 技术栈开发
*   必须支持 Windows 和 macOS 双平台
*   UI 风格为传统工具风格
*   必须具备历史记录功能

## 8. 项目边界 (Out of Scope)
*明确本阶段**不做**什么，避免需求蔓延。*
*   暂不支持全局快捷键快速打开工具
*   暂不支持暗色/亮色主题切换
*   暂不开发插件系统
*   暂不考虑 Linux 平台支持
*   暂不支持多语言国际化
