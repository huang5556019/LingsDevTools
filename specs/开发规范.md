## 4. TypeScript 特定规范

### 4.1 类型导入偏好

```typescript
// ✅ 正确：使用 import type
import type { HistoryItem } from './types';

// ❌ 错误：普通导入类型
import { HistoryItem } from './types';
```

### 4.2 interface vs type 使用场景

| 场景 | 偏好 | 原因 |
|------|------|------|
| **定义对象形状** | `interface` | 支持声明合并，扩展性更好 |
| **定义联合类型** | `type` | 更简洁 |
| **定义工具类型** | `type` | 更适合泛型操作 |
| **组件 Props** | `interface` | 便于后续扩展 |

```typescript
// ✅ 正确：组件 Props 用 interface
interface Base64ToolProps {
  defaultValue?: string;
  onValueChange?: (value: string) => void;
}

// ✅ 正确：联合类型用 type
type ToolType = 'encoder' | 'network' | 'data';

// ✅ 正确：工具类型用 type
type HistoryWithId = HistoryItem & { id: string };
```

### 4.3 避免的写法

```typescript
// ❌ 避免：使用 any
function processData(data: any) {
  return data;
}

// ✅ 替代方案：定义明确类型或使用 unknown + 类型守卫
interface Data {
  id: string;
  value: string;
}
function processData(data: Data) {
  return data;
}
```

```typescript
// ❌ 避免：非空断言
const item = historyItems[0]!;

// ✅ 替代方案：添加空值检查
const item = historyItems[0];
if (!item) return;
```

```typescript
// ❌ 避免：类型断言
const result = someValue as string;

// ✅ 替代方案：使用类型守卫
function isString(value: unknown): value is string {
  return typeof value === 'string';
}
if (isString(someValue)) {
  const result = someValue;
}
```

## 5. React + Electron 特定规范

### 5.1 组件设计原则

- **单一职责**：每个组件只负责一件事
- **可组合**：优先使用组合而非继承
- **无副作用**：组件应该是纯函数，副作用放在 useEffect 中
- **明确边界**：主进程逻辑与渲染进程逻辑严格分离

### 5.2 组件文件结构

```tsx
// 1. 类型导入
import type { Base64ToolProps } from './types';

// 2. 依赖导入
import { useState, useCallback } from 'react';

// 3. 内部类型定义
interface Base64ToolState {
  input: string;
  output: string;
}

// 4. 自定义 Hooks（如有）

// 5. 工具函数（如有）

// 6. 组件定义
export function Base64Tool({ defaultValue = '' }: Base64ToolProps) {
  const [state, setState] = useState<Base64ToolState>({
    input: defaultValue,
    output: '',
  });

  const handleEncode = useCallback(() => {
    // 实现编码逻辑
  }, []);

  return (
    <div>
      {/* 组件内容 */}
    </div>
  );
}
```

### 5.3 导入导出规则

- **导出方式**：优先使用命名导出
- **例外情况**：页面级组件可以使用默认导出

```typescript
// ✅ 正确：命名导出
export function Base64Tool() { /* ... */ }
export function UrlTool() { /* ... */ }

// 使用时
import { Base64Tool } from './encoder/Base64Tool';
```

### 5.4 导入排序

```typescript
// 按以下顺序分组，组间空一行：
// 1. React 相关
// 2. 第三方库
// 3. 内部模块（使用 @/ 别名）
// 4. 相对路径
// 5. 类型导入

import { useState, useCallback } from 'react';
import axios from 'axios';

import { useAppStore } from '@/store/useAppStore';

import { formatJson } from './utils';

import type { HistoryItem } from './types';
```

### 5.5 Electron IPC 使用规范

- **预加载脚本**：只暴露必要的 API，不要暴露完整的 Node.js API
- **频道命名**：使用 `namespace:action` 格式
- **错误处理**：IPC 调用必须有错误处理

```typescript
// ✅ 正确：预加载脚本 API
contextBridge.exposeInMainWorld('electronAPI', {
  saveHistory: (item: HistoryItem) => ipcRenderer.invoke('db:saveHistory', item),
  getHistory: (type: string) => ipcRenderer.invoke('db:getHistory', type),
});

// ✅ 正确：渲染进程调用
try {
  const history = await window.electronAPI.getHistory('http');
} catch (error) {
  console.error('Failed to get history:', error);
}
```

## 6. 样式规范

### 6.1 唯一样式方案

- **Tailwind CSS**：唯一的样式方案，禁止使用 CSS Modules 或 styled-components
- **动态类名处理**：使用 `clsx` 或 `cn` 工具函数
- **响应式设计**：使用 Tailwind 的响应式前缀

```tsx
// ✅ 正确：使用 Tailwind CSS
import { clsx } from 'clsx';

function Button({ variant = 'primary', children }: ButtonProps) {
  const buttonClass = clsx(
    'px-4 py-2 rounded font-medium',
    variant === 'primary' && 'bg-blue-600 text-white hover:bg-blue-700',
    variant === 'secondary' && 'bg-gray-200 text-gray-800 hover:bg-gray-300'
  );

  return <button className={buttonClass}>{children}</button>;
}
```

### 6.2 禁止的样式写法

```tsx
// ❌ 禁止：内联样式（特殊情况须注释理由）
<div style={{ color: 'red', fontSize: '14px' }} />

// ❌ 禁止：创建独立 CSS 文件
// import './styles.css';

// ✅ 正确：使用 Tailwind
<div className="text-red-600 text-sm" />
```

## 7. Git 工作流

### 7.1 分支策略

> 独立开发者采用简化 Trunk-Based 开发

```
main
  └── feat/add-base64-tool
  └── fix/history-save-error
```

| 规则 | 说明 |
|------|------|
| **主分支** | `main`，始终可部署 |
| **功能分支** | 从 `main` 创建，合并回 `main` |
| **分支生命周期** | 1-3 天 |
| **合并方式** | Squash Merge，保持 main 分支历史整洁 |

### 7.2 分支命名规范

```
<type>/<short-description>
```

| 类型 | 用途 | 示例 |
|------|------|------|
| `feat/` | 新功能 | `feat/add-base64-tool` |
| `fix/` | Bug 修复 | `fix/history-save-error` |
| `refactor/` | 重构 | `refactor/db-layer` |
| `docs/` | 文档更新 | `docs/update-readme` |
| `chore/` | 构建/配置 | `chore/upgrade-electron` |

### 7.3 提交信息规范 (Conventional Commits)

所有 Git 提交信息**必须**遵循 [Conventional Commits](https://www.conventionalcommits.org/) 格式：

```
<type>(<scope>): <subject>

[body]

[footer]
```

#### 类型 (type)

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变功能） |
| `style` | 代码格式调整 |
| `docs` | 文档变更 |
| `test` | 测试相关 |
| `chore` | 构建/工具/依赖 |
| `perf` | 性能优化 |

#### 作用域 (scope)

| scope | 对应范围 |
|-------|---------|
| `encoder` | 编码解码工具包 |
| `network` | 网络工具包 |
| `data` | 数据处理工具包 |
| `history` | 历史记录管理 |
| `db` | 数据库层 |
| `ipc` | IPC 通信 |
| `ui` | UI 组件 |
| `build` | 构建配置 |

#### 提交信息规则
- `subject`：中文、小写开头、不加标点、50 字符以内
- `body`：解释**为什么**，不是**做了什么**
- `footer`：如有相关 Issue 可以引用

#### 示例

```
feat(encoder): add base64 encode/decode tool

实现 Base64 编码解码功能，支持实时转换和结果复制
```

```
fix(history): prevent duplicate history items

在保存历史记录前检查是否已存在相同内容，避免重复
```

## 8. 代码注释规范

### 8.1 何时写注释

| 需要注释 | 不需要注释 |
|----------|-----------|
| 解释"为什么"这样做 | 解释"做了什么" |
| 复杂的业务逻辑 | 简单明了的代码 |
|  workaround 或临时方案 | 自解释的代码 |
| 非显而易见的边界情况 | 标准的 React/Electron 模式 |

### 8.2 注释风格

```typescript
// ✅ 解释 "为什么"
// 使用 btoa 而不是 Buffer，因为渲染进程中更简单
const encoded = btoa(input);

// ❌ 描述 "做了什么"
// 对输入进行 Base64 编码
const encoded = btoa(input);
```

```typescript
// TODO: 添加输入验证，2024-04-15

// HACK: 临时绕过 Electron IPC 限制，后续重构
```

## 9. 错误处理规范

### 9.1 主进程错误处理

```typescript
// 统一的错误处理
ipcMain.handle('db:saveHistory', async (_, item: HistoryItem) => {
  try {
    const result = await database.save(item);
    return { success: true, data: result };
  } catch (error) {
    console.error('Failed to save history:', error);
    return { success: false, error: '保存失败，请重试' };
  }
});
```

### 9.2 渲染进程错误处理

- 所有 IPC 调用必须有 try-catch
- 网络请求必须有超时和重试机制
- 用户友好的错误提示，不暴露技术细节

```typescript
// ✅ 正确：完整的错误处理
try {
  const response = await window.electronAPI.saveHistory(item);
  if (!response.success) {
    alert(response.error);
    return;
  }
  // 处理成功
} catch (error) {
  console.error('Save history error:', error);
  alert('操作失败，请稍后重试');
}
```

### 9.3 安全原则
- 永远不在 UI 中暴露原始错误堆栈或数据库错误信息
- 不要将用户输入直接传递给 SQLite 查询，使用参数化查询

## 13. AI 协作协议

> 本节定义 AI 辅助开发时必须遵守的规则，确保 AI 生成的代码与人工代码保持一致性。

### 13.1 AI 写代码前必读

| 文档 | 目的 |
|------|------|
| `specs/产品概述.md` | 理解项目目标和核心功能 |
| `specs/技术栈.md` | 确认技术选型和工具链 |
| `specs/项目结构.md` | 了解目录结构和模块划分 |
| `specs/开发规范.md` | 遵守代码规范和约定 |
| 相关模块的现有代码 | 理解当前实现，保持一致性 |

### 13.2 AI 写代码的规则

| 规则 | 说明 |
|------|------|
| **读后写** | 修改文件前必须先读取该文件 |
| **最小改动** | 只修改任务要求的部分，不做"顺便优化" |
| **不猜版本** | 安装依赖时使用 `pnpm add`，不手写版本号 |
| **不引入新依赖** | 除非任务明确要求 |
| **不写无意义注释** | 不写复述代码的注释 |
| **不创建多余文件** | 不主动创建文档或配置文件 |
| **遵循现有模式** | 相同类型的代码必须遵循已有文件的写法 |
| **提交信息规范** | 严格遵循 Conventional Commits |

### 13.3 AI 写完代码后必做

| 步骤 | 操作 |
|------|------|
| **1. 类型检查** | 运行 `tsc --noEmit` 确认无编译错误 |
| **2. Lint 检查** | 运行 `eslint` 确认无 Lint 错误 |
| **3. 依赖方向** | 确认没有违反目录结构中的依赖方向原则 |

### 13.4 错题本机制

当 AI 生成的代码出现问题时，在 `docs/ai-mistakes.md` 中记录：

| 记录项 | 说明 |
|--------|------|
| **错误类型** | 类型错误 / 逻辑错误 / 规范违规 / 依赖错误 |
| **错误描述** | 具体出了什么问题 |
| **根因分析** | 为什么会犯这个错 |
| **正确做法** | 下次应该怎么做 |

## 15. 安全编码规范

| 类别 | 规则 |
|------|------|
| **输入校验** | 所有用户输入必须在客户端和主进程双重校验 |
| **SQL 注入防护** | 所有 SQLite 查询必须使用参数化查询，禁止字符串拼接 |
| **XSS** | 渲染用户输入内容时必须进行转义，使用 `textContent` 而非 `innerHTML` |
| **密钥管理** | 不硬编码任何密钥，敏感数据存储在用户数据目录 |
| **IPC 安全** | 预加载脚本只暴露必要的 API，不暴露完整 Node.js 能力 |
| **文件操作** | 限制文件操作范围，只访问用户数据目录 |
